# Restormel Site Pages Outline and Content Requirements

## Purpose
Define the core pages for `restormel.dev`, their strategic role, and the content required to keep the site commercially coherent.

## Site goals
The site must help users:
1. understand what Restormel is
2. see why it is different from tracing and generic RAG tooling
3. try a useful workflow quickly
4. understand how it integrates with their existing stack
5. choose a self-serve or enterprise path

## Primary audience lanes

### Lane 1: Developers and AI engineers
Need clear docs, examples, import paths, and APIs.

### Lane 2: Evaluation and product teams
Need evidence inspection, comparison, and governance workflows explained plainly.

### Lane 3: Technical buyers
Need security, procurement, and commercial clarity.

## Core site map
- Homepage
- Product: Reasoning Workspace
- Product: Graph-aware Evals
- Product: Lineage Exports
- Playground
- Docs
- Pricing
- Security / Trust
- About
- SOPHIA / Reference App

## Page requirements

### Homepage
Must answer in one screen:
- what Restormel is
- what it shows that traces alone do not
- what the first product is
- how to try it now

### Product: Reasoning Workspace
Should show:
- support inspection
- contradiction inspection
- provenance trails
- compare mode
- why this is better than reading logs

### Product: Graph-aware Evals
Should explain:
- reasoning-quality evaluation
- support sufficiency
- contradiction exposure
- retrieval structure diagnostics
- regression comparison

### Product: Lineage Exports
Should explain:
- governance-grade decision lineage
- audit artefacts
- evidence-backed summaries
- retained run history

### Playground
Should include:
- sample runs
- import flow
- one-click compare demo
- example lineage export

### Docs
Should prioritise:
- contracts
- adapters
- APIs
- data import
- evaluator model
- examples from SOPHIA

### Pricing
Should package:
- hosted convenience
- retention
- comparison history
- evaluator usage
- team collaboration
- enterprise controls

### Security / Trust
Should cover:
- data handling
- retention
- access controls
- enterprise readiness
- deployment options where relevant

## Editorial rules
- product copy should lead with use cases, not infrastructure
- do not describe GraphRAG as the headline product category
- explain integrations as adapters, not platform sprawl
- avoid generic “AI platform” wording
